## Submitting a run

From the editor, `Run` navigates to `/workflows/{slug}/runs/new`. The page
reads the workflow's latest version (`GET /workflows/{slug}`) and renders
**one `AutoForm` per input node** ‚Äî the graph itself lists every required
value. Struct-typed inputs expand into nested field groups rendered by the
same AutoForm widget.

On submit the values are keyed by InputNode `node_key` and POSTed to
`/workflows/{slug}/runs`. The page then redirects to
`/workflow-runs/{id}` for live polling.

```bash
curl -X POST https://api.mecapy.com/workflows/bolted-joint/runs \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": {
      "bolt_input": {
        "d": 12,
        "p": 1.5,
        "As": 84.3,
        "Re_min": 640,
        "quality_class": "8.8"
      },
      "assembly_input": {
        "dh": 18,
        "do": 13,
        "Rc": 210,
        "mu_p_min": 0.1
      },
      "loads_input": { "FA_max": 15000, "Ft_max": 3500 },
      "tightening_input": {
        "mu_tot_min": 0.08, "mu_tot_max": 0.14, "precision_class": "A"
      }
    }
  }'
```

The response carries the **freshly-frozen execution plan** and the initial
`node_states` ‚Äî every function node in `pending`. Input nodes do not appear
in `node_states`: they are pre-resolved at submission and injected directly
via `input_node` bindings on each downstream step. No worker jobs have been
submitted yet: the run starts advancing on the first `/tick`.

The submitted values are echoed back on `run.inputs` so the detail page can
render an "Submitted inputs" card without a separate lookup.

<Note>
  **Required-ness.** An input node is required (and cannot be omitted in
  `inputs`) iff `input_spec.required === true`. Missing required keys
  return `422 { "detail": { "missing_required_inputs": [...] } }`.
</Note>

## The tick loop

A run is a state machine driven by one HTTP endpoint:

```text
POST /workflow-runs/{id}/tick   ‚Üí   WorkflowRunResponse
```

One call performs three phases atomically (one DB commit):

1. **Reconcile running nodes** ‚Äî poll Redis for every node currently
   `running`. Write outputs on success, record errors on failure.
2. **Submit ready nodes** ‚Äî every `pending` function node whose upstream
   function dependencies are all `success` has its inputs resolved and is
   pushed to the worker queue. Input nodes do **not** count as
   dependencies ‚Äî they are always ready.
3. **Finalise** ‚Äî if a terminal condition is reached, update the
   run-level status (`completed` / `failed` / `cancelled`).

The endpoint is **idempotent** ‚Äî calling it with no external progress
(no job has completed in Redis since the last call) simply returns the
same state.

### Who calls /tick?

The platform deliberately does **not** run a background orchestrator.
The runtime needed to drive a run comes from one of:

- **The frontend run page** ‚Äî polls every 2 s while the run's status is
  `pending` or `running`, stops on terminal.
- **A cron** or any periodic job ‚Äî for unattended runs.
- **The tests** ‚Äî pytest mocks Redis and drives ticks synchronously.

<Note>
  This architecture is deliberately robust to API restarts. If the
  Uvicorn worker goes down mid-run, no in-flight state is lost ‚Äî the
  Postgres `workflow_runs.node_states` JSONB column is the source of
  truth. The next `/tick` picks up where the last one left off.
</Note>

### Input bindings in `plan_snapshot`

Each step's `input_bindings` map every declared function input port to one
of three source types:

- `{"source": "input_node", "input_key": "bolt_input", "from_port": "value"}`
  ‚Äî read `run.inputs["bolt_input"]` verbatim (primitive, list, or struct
  alike).
- `{"source": "edge", "from_node_key": "X", "from_port": "y"}` ‚Äî read
  `node_states["X"].outputs["y"]`. Used between chains.
- `{"source": "chain", "from_node_key": "X", "from_port": "y"}` ‚Äî same as
  `edge` but the source lives in the same chain (fusion hint for v1.5
  worker chain-exec).

There is no "free" source anymore ‚Äî strict mode guarantees every port is
covered.

## Fail-fast semantics (FRO-wkf-07)

When a function node fails:

- Every **pending** function node downstream of the failure is marked
  `cancelled` ‚Äî no new job is submitted for them.
- **Running** nodes on independent branches keep running to completion
  ‚Äî the spec deliberately lets parallel work finish to avoid wasted
  compute.
- The run itself flips to `failed` with `first_failed_node_key` set.

```text
   A (failed)
   ‚îÇ
   ‚îú‚îÄ‚îÄ B (cancelled)        ‚Üê was pending, nothing submitted
   ‚îÇ
   ‚îî‚îÄ‚îÄ C (cancelled)        ‚Üê was pending
       ‚îÇ
   X ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ E (running)     ‚Üê independent branch keeps going
   ‚îÇ
   ‚îî‚îÄ‚îÄ F (cancelled after C)
```

## Cost model ‚Äî the cold-start caveat

Each function node pays one Python cold start when submitted. Empirically
on Scaleway Serverless containers: ~6 s for a noop, ~9 s end-to-end
(FRO-perf-01). A naive four-node chain would therefore take ~24 s
minimum ‚Äî which would make workflows dead-on-arrival for real use.

Input nodes add **zero** cold-start cost ‚Äî they are resolved in-process
during submission.

v1 mitigates this with one optimisation already wired in:

<Note>
  **Chain fusion (FRO-wkf-09)** ‚Äî the plan builder greedily groups
  consecutive linear function nodes that share the same `(package_id,
  package_version_id)` into a single **chain**. In v1.5 with
  chain-exec in the worker, the whole chain runs in one container ‚Äî
  one cold start instead of N.

  In v1, chain fusion **still reduces** the wall-clock time because
  each step's inputs are resolved locally from the previous step's
  outputs without an extra round-trip, but each step still pays its
  own cold start. Plan your graphs accordingly:

  - Two consecutive same-package steps ‚âà 2√ó cold start (~12 s).
  - Four distinct packages in sequence ‚âà 4√ó cold start (~24 s).
  - A branchy DAG with 3 parallel branches of 2 steps each ‚âà 4√ó cold
    start (bottleneck = longest path).
</Note>

### Parallelism

Independent branches are submitted **simultaneously** on the first
`/tick` after they become ready (FRO-wkf-08). The platform does not
serialise on arbitrary work order ‚Äî a diamond DAG
`A ‚Üí {B,C} ‚Üí D` runs `B` and `C` in parallel, `D` starts after both
finish.

## Terminal outputs

A workflow run's "result" is the outputs of **terminal function nodes** ‚Äî
function nodes that no other node consumes. Input nodes are never
terminal. For the diamond above, `D`'s outputs are the run's
`terminal_outputs`. Multiple sinks produce multiple terminal output groups
keyed by `node_key`:

```json
{
  "terminal_outputs": {
    "safety_check": { "verdict": "OK", "margin": 2.3 },
    "bom_export":   { "items": [‚Ä¶] }
  }
}
```

Intermediate node outputs remain inspectable in `node_states[key].outputs`
but are **not** aggregated into `terminal_outputs` ‚Äî the contract is
explicit leaves only.

## Observability

During a run:

- **`GET /workflow-runs/{id}`** ‚Äî single fetch, same shape as `/tick`
  without advancing state.
- **`GET /workflow-runs/{id}/plan`** ‚Äî the frozen execution plan and
  current node states (audit / replay).
- **Run detail page** ‚Äî visual DAG at `/workflow-runs/{id}`, coloured
  by status (pending grey, running blue pulsing, success green,
  failed red, cancelled amber). Also displays a "Submitted inputs" card
  listing `run.inputs` for audit, and a "Terminal outputs" card on
  completion.
- **Runs list** ‚Äî `/workflow-runs` with status filters for batch
  triage across all your runs.

## Not in v1

- **Retry policy / exponential back-off** on failed nodes.
- **Partial re-run** ‚Äî continue from a specific node after a fix.
- **Run cancellation** ‚Äî no `DELETE /workflow-runs/{id}` endpoint yet.
- **Worker chain-exec** ‚Äî one cold start per chain (FRO-wkf-12 v1.5).
- **Warm pool** ‚Äî top-N pre-warmed containers (FRO-wkf-12 v1.5).
